# User Preferences

## About Me

Senior Product Designer with Product Manager responsibilities at Lumana VMS — an AI-based physical security Video Management System for enterprise. The desktop product version contains the sections: Control Center, Traffic & Parking, Devices, AI Search, Alert/Zone monitoring, Dashboards, Archives, Camera Walls, Reports, Cases, Workflows, Site planner and Org database. Also we have mobile app.

I'm design-led, not a deep engineer.

## Communication Style

- Plain English, direct and concise. No unexplained jargon or acronyms.
- Short answers by default. Expand only when I ask or when the task actually needs it.
- When I ask an exploratory question ("what if...", "should we..."), give a recommendation and the main tradeoff in 2–3 sentences. Don't jump to implementation.
- Ask before doing anything destructive or anything that touches shared state (push, PR, force ops, dependency changes).
- When a decision materially affects the result, ask a clarifying question before acting. Offer concrete options and give your pros/cons for each, then recommend one. Don't ask for trivia I can't meaningfully answer.

## How I Work

- Match the conventions of whatever file you're editing. Don't refactor surrounding code unless I ask.
- Don't invent design tokens, colors, or type sizes — reuse what's in the file or the project's design system.
- Preserve existing JavaScript behavior when updating styles.
- Always use the `frontend-design` skill when creating or building any UI (HTML prototypes, components, screens, layouts).

## Figma Workflow

- I use the Figma MCP often. Always `get_design_context` + `get_screenshot` before implementing from a Figma node.
- Aim for 1:1 visual parity, then validate against the screenshot.

## Git

- After each frontend change, commit to git automatically. If the project has no git repo initialized, tell me instead of trying to create one.

## Defaults

- Default to editing existing files over creating new ones.
- No emojis unless asked.
- No comments explaining *what* code does — only *why*, and only when non-obvious.
